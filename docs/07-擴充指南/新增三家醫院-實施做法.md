# 新增三家醫院：實施做法建議

> 這份文件回答的是「**要一次新增三家醫院，如果是我會怎麼做？**」——著重策略、順序與風險取捨。
> 逐點的檔案修改清單請看 [新增醫院](新增醫院.md)；本文不重複那張表，而是說明「怎麼安排這件事最穩」。

---

## 1. 先看清楚問題本質

目前三家醫院（成大 / 奇美 / 郭綜合）是**硬編碼**散落在約 18 個位置：`switch`、連續 `if`、固定清單、Excel 分頁、JSON 資料檔。詳見 [新增醫院 §3 修改清單](新增醫院.md)。

一次新增三家，代表這 18 個點每個都要再加三筆 → 實際是 **18 × 3 ≈ 50+ 處**手動修改。最大的風險不是「改不動」，而是：

- **靜默漏算**：`DashboardService.GetHospitalName` 與 `SubjectNoHelper.GetHospital` 是連續 `if`，漏改**不會報錯**，但該院病人會從儀表板長條圖或抽血檔名比對中悄悄消失，很難察覺。
- **編號重號**：`SubjectNoGenerator.json` 的鍵名若與模型序列化名對不上，會從 0 起算造成重號。
- **隨機表快取**：`RandomListRuntime.json` 是執行期快取，新分頁若沒清快取不會生效。

一次改三家，這些陷阱各要中三次。**這正是我會先停下來想「要不要換做法」的地方。**

---

## 2. 兩種做法與取捨

### 做法 A：照清單土法煉鋼 ×3
直接依 [新增醫院](新增醫院.md) 的清單，把三家醫院當三次獨立新增逐點加上去。

- ✅ 不動既有架構，風險侷限在「加資料」層面，最貼近原作者寫法。
- ❌ 50+ 處手改、重複勞動，靜默漏算的機率隨次數放大；日後第七家還要再來一次。

### 做法 B：先做「醫院清單集中化」小重構，再以資料驅動新增（**我的建議**）
先把散落的醫院定義收斂成**單一份醫院清單（registry）**，讓原本各處的 `switch`/`if` 改成「讀這份清單」，然後新增醫院就變成「在清單加三筆資料 + 準備 Excel/JSON 資料」。

- ✅ 把「未來每加一家都要改 18 處」變成「改 1 份清單」；**根除靜默漏算**（因為不再有平行的 `if` 需要同步）。
- ✅ 一次加三家的效益剛好覆蓋重構成本，且第七、八家幾乎零成本。
- ❌ 動到既有邏輯，需要較完整的驗證；必須嚴守「只集中、不改既有代碼字串」。

> **我的取捨**：因為是**一次三家**、且痛點是「重複 + 靜默漏算」，做法 B 的重構成本會被攤平、又能消除最危險的陷阱，屬於「順手把正在動的地方改好」的合理範圍，不是無關重構。若只加一家，我會傾向做法 A。

---

## 3. 建議做法（做法 B）的執行順序

核心原則：**先建立清單與資料 → 讓邏輯改讀清單 → 最後補畫面 → 全程只新增、不改既有字串。**

### 步驟 0：先蒐集資料（動手前必須拿到）
新增醫院有一半是「資料」而非「程式」，缺料無法完成。動工前先向試驗團隊確認每家醫院的：

| 需要的資訊 | 用途 |
| --- | --- |
| 英文代碼（如 `XXX`）＋院別全名＋短名 | 受試者編號前綴、下拉、儀表板 |
| 受試者編號起始流水號 | `SubjectNoGenerator.json` |
| 隨機分組表（Early / Advance，EC / OC） | `RandomList.xlsx` 新分頁的實際隨機碼 |
| 抽血檢驗項目是否與既有相同 | 決定要不要新增 `抽血檢驗血液/生化N.json` |
| 是否調整總目標收案人數（現為 294） | 儀表板完成率分母 [DashboardViewModel.cs:50](../../src/CTMS/CTMS.DataModel/Models/Dashboard/DashboardViewModel.cs) |

⚠️ 隨機碼是臨床試驗的關鍵資料，必須由試驗統計/PI 正式提供，**不可自行編造**。

### 步驟 1：建立單一醫院清單（重構核心）
在 [MagicObjectHelper.cs](../../src/CTMS/CTMS.Share/Helpers/MagicObjectHelper.cs) 或新開一個 `HospitalRegistry`，定義一份 `List<Hospital>`，每筆含：代碼、全名、短名、隨機表分頁名、抽血檔名後綴、流水號鍵名。**既有的 `NCKUH/CHIMEIH/KGH`、`成大/奇美/郭綜合` 常數保留不動**，讓清單引用它們，確保既有資料不失準。

再提供幾個查詢方法取代散落邏輯：`GetAll()`、`GetByPrefix(subjectNo)`、`GetByDisplayName(院別)`。

### 步驟 2：把散落的 switch / if 改成讀清單
逐一將 [新增醫院 §3](新增醫院.md) 清單中的判斷改為呼叫上述方法。優先處理兩個高風險點：
- `DashboardService.GetHospitalName` → 改用 `GetByPrefix`，消除靜默漏算。
- `SubjectNoHelper.GetHospital` / `GetBloodFilename` → 改用清單查詢。

其餘（`DropDownListDataService.Get院別`、`RandomListService` 讀分頁、`PatientService` 院別對應等）同法改為 `foreach (GetAll())`。

### 步驟 3：備妥資料檔
- `RandomList.xlsx`：為三家各加 Early / Advance 分頁，填入步驟 0 拿到的隨機碼（注意 `ReadSheet` 預設讀到第 2000 列）。
- `SubjectNoGenerator.json`：加入三家的流水號鍵（鍵名須與模型序列化名完全一致）。
- **刪除或重建 `RandomListRuntime.json` 快取**，否則新分頁不生效。

### 步驟 4：新增三家醫院資料
在步驟 1 的清單裡新增三筆。若步驟 2 做得徹底，這一步幾乎就完成了功能。

### 步驟 5：補畫面
- `RandomPage.razor`：為每家新增 EC/OC × Early/Advance 共 4 個分頁（三家共 12 個）。
- 如需後台維護流水號，`LastSubjectNoView.razor` 也要補欄位。

### 步驟 6：收尾
依專案規範 `appsettings.json` 版本號 Patch +1、更新 [新增醫院](新增醫院.md) 與相關文件。

---

## 4. 驗證（每家醫院都要跑一遍）

沿用 [新增醫院 §5 驗證檢查清單](新增醫院.md)，對三家各驗一次，重點確認：

- 下拉出現三家新院、收案後編號 `XXX0001` 逐次遞增、`SubjectNoGenerator.json` 同步。
- 隨機分組能從新分頁取碼並回填 Treatment。
- **`/Dashboard` 醫院長條圖出現三家新院且病例數正確**（這是靜默漏算最容易被抓出來的地方，務必逐家看）。
- 抽血檢驗抓到正確的檔名；CSV 單列匯出院名正確。

因為做法 B 動了既有邏輯，**除了新院，既有三家也要回歸驗證一次**（編號、隨機、儀表板分組都要照舊正確）。

---

## 5. 如果時間很趕、只能選做法 A

可以，但務必：
1. 把 [新增醫院 §3](新增醫院.md) 那張表當**逐項打勾清單**，三家 × 每一列都勾過。
2. 對「連續 `if` 的兩個高風險點」特別複查。
3. 驗證階段一定要逐家看儀表板長條圖，確認沒有靜默漏算。

土法煉鋼能動，但這筆技術債會留到下一次新增醫院時再付一次。
