# 資料流與分層

說明請求如何在各層流動，以及 CTMS 最關鍵的資料保存模式 —— **病患 `JsonData`**。

## 1. 分層責任

```
Razor 頁面 / View (CTMS/Components)
        │ 呼叫
        ▼
領域服務 Service (CTMS.Business)
        │ 讀寫
        ├──► EF Core 實體 (CTMS.EntityModel / BackendDBContext) ──► SQLite
        └──► 臨床資料模型 (CTMS.DataModel) ◄── JSON 序列化 ──► Patient.JsonData
```

- **UI 層（CTMS）**：Blazor 頁面與 View 負責呈現與互動，透過 DI 取得服務。
- **商業邏輯層（CTMS.Business）**：所有領域服務（Dashboard、BloodExame、AIIntegrate、Survey、SubjectNoGenerator…）集中於此，是頁面與資料之間的唯一通道。
- **資料模型層（CTMS.DataModel）**：定義臨床資料的 C# 物件結構，與資料庫無關，主要被序列化進 `JsonData`。
- **實體模型層（CTMS.EntityModel）**：EF Core 實體與 `BackendDBContext`，對應 SQLite 資料表。

## 2. 兩種資料保存策略（重要）

CTMS 同時使用兩種存法，理解這點是看懂整個系統的關鍵：

### (A) 關聯式資料表（結構穩定的主檔）

少數核心實體以正規化資料表保存，作為 `BackendDBContext` 的 DbSet：

- `Patient`（受測者主檔）
- `MyUser`（使用者）
- `Project`（專案）、`RoleView`（角色）、`RoleViewProject`（角色-專案關聯）
- `OperationHistoryTrace`（操作歷程）

這些是查詢、清單、權限與儀表板統計的依據。

### (B) JsonData 文件式保存（多變的臨床資料）

每位受測者**大量且結構多變的臨床資料**（臨床資訊、手術、病理、化療、抽血、副作用、問卷、追蹤、AI 結果…）並不各自建表，而是序列化成 JSON 字串，存進 `Patient.JsonData` 欄位。

- 對應的 C# 結構定義在 `CTMS.DataModel`（以 `PatientData` 為根，內含 `臨床資訊` 與 `臨床資料`）。
- 頁面操作 → 服務讀出 `JsonData` 反序列化成物件 → 修改 → 再序列化寫回 `Patient.JsonData`。
- 好處：臨床欄位增修不需 migration；代價：跨個案的欄位查詢/匯出需在程式端攤平（見匯出大表）。

> `JsonData` 的詳細結構與欄位，見 [../03-資料模型/領域資料模型.md](../03-資料模型/領域資料模型.md)。

## 3. 典型請求流程（以個案資料維護為例）

1. 使用者於 `/Browser` 選擇受測者，以 `Patient.Code` 進入個案頁。
2. 進入 `/BasicClinical/{code}` 或其他模組頁，頁面 `OnInitialized` 透過服務以 `code` 載入 `Patient`，反序列化 `JsonData` 成 `PatientData`。
3. 各模組 View 綁定 `PatientData` 的對應區塊（如抽血 → `BloodTest抽血檢驗血液`）。
4. 使用者編輯，前端做欄位驗證與條件顯示（詳見各模組第 5 章）。
5. 儲存時服務把修改後的物件序列化寫回 `Patient.JsonData`，`SaveChanges` 入庫。
6. 視操作類型寫入 `OperationHistoryTrace`。

## 4. Visit Code（訪視）模式

許多臨床資料是**縱向（longitudinal）**的：同一類資料在不同訪視時點各有一筆。系統以 **Visit Code** 區分時點，讓抽血、副作用、追蹤等資料能依時序維護。匯出大表時，這些訪視會被攤平成欄或多列。

## 5. AI 資料流（跨主站與背景服務）

1. 主站在個案頁上傳 DICOM 影像或標註修正，寫入上傳/佇列資料夾。
2. 背景 `AIAgent` 輪詢佇列，依 Phase1/2/3 執行 DICOM→PNG 與 AI 推論。
3. 推論輸出（骨骼肌指標 CSV、結果影像）回寫到輸出資料夾與個案資料。
4. `/RiskAssessment/{code}` 讀出結果顯示，醫師簽名寫回。

詳見 [../05-AI推論/overview.md](../05-AI推論/overview.md) 與 [../05-AI推論/AIAgent背景服務.md](../05-AI推論/AIAgent背景服務.md)。

## 6. 物件對應（AutoMapper）

Entity ↔ DTO/ViewModel 的轉換使用 AutoMapper（`CTMS.Business`）。頁面通常操作 DTO/ViewModel，由服務負責與實體互轉。
