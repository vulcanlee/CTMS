# 系統總覽

> 本文件是整套 CTMS 文件的入口導讀，先讓人類開發者與 AI 助手在五分鐘內掌握「這個系統在做什麼、為誰做、由哪些部分組成」。細節請依各章連結深入。

## 1. 這是什麼系統

**CTMS（Clinical Trial Management System，臨床試驗管理平臺）** 是一套以**婦科癌症（子宮內膜癌 EC、卵巢癌 OC）臨床試驗**為核心情境的管理平臺。系統名稱在設定檔中為「AI 臨床試驗管理平臺」（見 [src/CTMS/CTMS/appsettings.json](../../src/CTMS/CTMS/appsettings.json)）。

它圍繞「**單一受測者個案**」展開工作，把臨床試驗中分散在紙本、Excel、問卷、影像、醫囑的資料集中到同一平臺，並把 **AI 骨骼肌風險評估**流程直接嵌入研究現場。

## 2. 解決什麼問題

- **資料分散** → 受測者所有資訊集中於一個平臺，圍繞個案完整追蹤。
- **資料不一致** → 固定資料模型、下拉選單、Visit Code 時序分段、角色權限控管。
- **研究治理** → 帳號申請審核、操作歷程稽核、受測者狀態切換、醫師簽名、隨機分派與 Subject No 編號。
- **AI 落地** → 上傳影像 → AI 推論 → 量化指標與風險評估 → 實驗組/對照組 → 醫師簽名與歷程。
- **跨院協作** → 以成大、奇美、郭綜合等合作院所的收案與分派情境設計。

## 3. 使用者角色

| 角色 | 主要工作 |
| --- | --- |
| 臨床研究護理師 / CRC | 收案、維護臨床資料、抽血、副作用、問卷 |
| 研究主持人（PI）與協同醫師 | 監看進度、確認資料 |
| 放射科醫師 | 影像標註、AI 結果簽名確認 |
| 婦產科醫師 | 臨床決策、AI 結果簽名確認 |
| 系統管理者 | 帳號/角色/專案/隨機表/操作歷程管理 |
| 維運 / 開發人員 | 部署、設定、資料庫維護 |

## 4. 三家合作醫院

目前系統內建三家醫院，未來預計擴充。涉及醫院清單的程式碼散布在儀表板統計、受試者編號、隨機分組等處；新增醫院時請依 [擴充指南：新增醫院](../07-擴充指南/新增醫院.md) 逐項修改。

## 5. 系統由哪些部分組成

CTMS 不是單一 Web 專案，而是 .NET 9 的多專案解決方案（`src/CTMS/CTMS.sln`），主要分成兩個可執行程式與多個函式庫：

- **CTMS**（Blazor Server Web App）— 臨床試驗管理主網站，所有 UI 與頁面。
- **AIAgent**（Worker Service）— 背景服務，輪詢佇列資料夾執行 DICOM/AI 推論管線。
- 其餘 7 個函式庫分別承載商業邏輯、資料模型、實體模型、Excel 工具等。

詳見 [解決方案架構](../02-架構/解決方案架構.md)。

## 6. 主要流程鳥瞰

1. 登入（`/Auths/Login`）→ 首頁 splash（`/`，初始化隨機表與編號產生器）→ 儀表板（`/Dashboard`）。
2. 於受測者瀏覽頁（`/Browser`）找到或新增個案。
3. 進入個案總控台（`/BasicClinical/{code}`），由底部導覽切換臨床資料、抽血、副作用、問卷、追蹤、風險評估等模組。
4. 各模組依 Visit Code 維護縱向資料，結果寫回病患 `JsonData`，部分操作產生操作歷程。
5. AI 流程：上傳影像/標註 → AIAgent 背景推論 → 結果回寫 → `/RiskAssessment/{code}` 顯示 → 醫師簽名。

> `{code}` 對應 `Patient.Code`，不是畫面顯示的 Subject No。
